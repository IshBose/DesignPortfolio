Background

Terraform Cloud has the concept of software modules as a foundational element for managing cloud infrastructure. Modules are reusable, self-contained packages of Terraform configurations that encapsulate a set of resources. They help organize and manage infrastructure as code by promoting reusability and maintainability. You can define a module once and then use it many times, which helps for the overall notion of DRY (Do not Repeat Yourself) code. Modules come in two core flavors - public, which are downloaded from the Terraform registry, and private, which are managed by internal organizations.

So what's wrong?

Modules have a lifecycle that follows most software processes. Companies want to gradually upgrade their software over time through versioning, think iOS 17 to iOS 18, and terraform modules are no different. Organizations will usually create multiple versions of a module in order to stay up to date with latest security standards, bug fixes, etc.

The core issue with this is that incentivising the end user to upgrade from one module version to the next is often tricky. Two different individuals who have two different roles currently have a gap in their Terraform experience, lets dig in further.

How do customers solve this problem today?

There are two core ways to approach this problem currently, which was told by customers. Each have downfalls and are indicative of the level of cloud maturity of the customer.

Empathy building

The core way my EPD team approached this was considering this gap between the module author persona, and the module consumer persona. To create empathy with the problem at hand, we sat in on many customer calls to understand how organizations were truly segmented.

A naive assumption I had was that the module author persona and the module consumer persona were likely going to be on the same team within a company, likely one focused on DevOps or cloud infra, but this couldn’t be further from the truth! Most module author personas were platform engineers on the central SRE team, while the module consumers were app developers or SWE’s looking to self serve infra quickly. This helped to further humanize the situation of communication and how it could sprawl quickly. These individuals likely never met face to face or had meetings to discuss these things, which made it more difficult to overcome.

Problem statement

Module authors need a native way in Terraform to communicate to module consumers for situations when you need to upgrade or migrate your module.

User flows

These customer calls and the problem statement helped us to devise some basic user flows. The parts of the flow that was highlighted are "new" areas.
